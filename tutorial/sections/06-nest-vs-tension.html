Nest vs tension

BSVA runs on Nestr. Every substantive task becomes a Nestr node — which kind depends on whether you're crossing a role boundary.

The rule (memorize)

Nest
Own work inside a role you fill

Your role is accountable, you don't need anyone else's time or authority. Past-tense title. Acceptance criteria in the description. users: [you].

Tension
Ask crossing role boundaries

Another role is accountable for the outcome or must approve. Uses Nestr's 5 processing pathways (request info / share info / request outcome / request action / governance).

Decision tree

Does work fall inside a role I fill?
✅ YES + no cross-role need → nest
⚠ YES but needs other role's review → nest (mine) + tension (for the cross-role piece)
➜ NO, someone else is accountable → tension to that role
Structural change (new role / policy / domain)?
→

Three worked cases

DevRel — writing a blog post

"I want to publish a blog on BRC-100."

  • Nest under DevRel role (your own work).
  • + Tension to Tech's sdk-maintainer role (for crypto-section fact-check).
  • + Tension to Marketing (for cross-post) — clustered into one weekly Marketing tension.

Tech — refactoring a service

"I need to widen the header-chain reorg window."

  • Nest under your role in Tech.
  • + Tension to Security if the change touches crypto / consensus (hard rule).

Biz-Dev — building a lead list

"I want to build a Tier-1 VC list for blockchain infra."

  • Nest under Biz-Dev role (own work, public sources).
  • No tensions — no one else's time is needed.

Cluster, don't spam

Nestr convention

An agent creating 20 tensions per hour is technically correct but practically hostile. BSVA norm: ≤ 2 tension submissions per day per target role unless urgent. Cluster related asks into one well-structured tension. The global:nestr-cluster skill does this automatically.
